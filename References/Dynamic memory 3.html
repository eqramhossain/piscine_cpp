1) Memory diagrams: what 

new

 and 

delete

 really do


Think of your program as having two big memory zones:

  • Stack: automatic, temporary, tied to scope

  • Heap: manual, persistent until you free it


Example 1: single 

int

int *p = new int(42);

What happens:

  1. new int(42) asks the heap for enough memory for one int

  2. That memory is initialized with 42

  3. The address of that memory is stored in p


A simple picture:

Stack                          Heap
-----                          -----
p  ------------------------->  [ 42 ]

So p itself is a normal pointer variable on the stack, but it points to memory on the heap.

Example 2: deleting it

delete p;

Now the heap memory is given back.

Stack                          Heap
-----                          -----
p  ------------------------->  [ freed ]

Important detail: p still contains an address after delete, but that address is now invalid. That is why using *p after delete is dangerous.

Example 3: dynamic array

int *arr = new int[3];

Picture:

Stack                          Heap
-----                          -----
arr ----------------------->   [ ? ][ ? ][ ? ]

You can use it like this:

arr[0] = 10;
arr[1] = 20;
arr[2] = 30;

Now:

Stack                          Heap
-----                          -----
arr ----------------------->   [10 ][20 ][30 ]

To free it:

delete[] arr;

That [] matters. Array allocation must be matched with array deallocation.

2) The most important rule: match allocation and deallocation


This is one of the biggest beginner traps.


Single object

int *p = new int(5);
delete p;

Array

int *arr = new int[5];
delete[] arr;

Why this matters


new and delete work as a pair.

new[] and delete[] work as a pair.


If you mix them, the program has undefined behavior. That means anything can happen: crash, corruption, weird bugs, or appearing to work until it does not.

3) Connection with C: 

malloc

 / 

free


This is a very useful bridge if you already know C.


C style

int *p = malloc(sizeof(int));
*p = 42;
free(p);

C++ style

int *p = new int(42);
delete p;

The idea is similar:

  • both allocate memory dynamically

  • both require manual release


But there is a major difference.

malloc

 does not construct objects


malloc only reserves raw bytes.


It does not call constructors.


new

 does both:

  • allocates memory

  • constructs the object


That is why new is the proper C++ tool for objects.


Example:

class Student {
public:
    Student() {
        std::cout << "constructor\n";
    }
};

Student *s1 = (Student *)malloc(sizeof(Student)); // not good in C++
Student *s2 = new Student();                      // correct C++

With new Student(), the constructor runs.

With malloc, it does not.

free

 does not destroy objects


free() releases memory, but it does not call destructors.


That is a huge difference in C++.


If a class has a destructor, you need delete so that the destructor runs.


Example:

class Test {
public:
    ~Test() {
        std::cout << "destructor\n";
    }
};

Test *t = new Test();
delete t;   // destructor is called

If you used free(t) instead, the destructor would not run, which breaks C++ object lifetime rules.

4) Why 

delete

 is more than “just free”


For a single object, this:

delete p;

really does two things:

  1. calls the destructor of the object

  2. releases the memory


That means delete is responsible for both cleanup and deallocation.


For arrays:

delete[] arr;

it destroys every element in the array, then frees the block.

5) Common mistakes to learn from


Mistake 1: using memory after delete

int *p = new int(7);
delete p;
std::cout << *p;   // dangerous

After delete, p is a dangling pointer. It points to memory that no longer belongs to you.

Mistake 2: forgetting 

delete

void f() {
    int *p = new int(9);
}

The function ends, the pointer variable disappears, but the heap memory is still allocated. That is a memory leak.

Mistake 3: wrong delete form

int *arr = new int[4];
delete arr;     // wrong

Must be:

delete[] arr;

Mistake 4: double delete

int *p = new int(3);
delete p;
delete p;   // very bad

You are freeing already freed memory. That is undefined behavior.

6) Step-by-step debugging exercises


These are good mental checks. Do not rush them. Trace the memory in your head.

Exercise A

int *a = new int(5);
int *b = a;
delete a;

Questions to ask yourself:

  • What does b point to?

  • Is that memory still valid?

  • What happens if you do *b now?


Think carefully: b still holds the same address, but the memory was freed. So b becomes dangerous too.

Exercise B

int *arr = new int[3];
arr[0] = 1;
arr[1] = 2;
arr[2] = 3;
delete[] arr;

Questions:

  • Is this correct?

  • Why is delete[] required here?


Yes, this is correct, because the memory was allocated as an array.

Exercise C

int *p = new int(10);
delete p;
p = NULL;

Questions:

  • Why set p to NULL after delete?

  • What danger does that reduce?


It reduces the chance of accidentally using a dangling pointer again.


In modern C++, nullptr is preferred over NULL:

p = nullptr;

Exercise D

int *p = new int(10);
int *q = new int(10);

delete p;
delete q;

Questions:

  • Are both objects deleted correctly?

  • Do p and q refer to the same memory?


Yes, this is fine because each pointer owns its own allocation.

7) A very important idea: ownership


This is a concept that will help you a lot later.


When you do:

int *p = new int(42);

you are now the owner of that memory.

That means you are responsible for deleting it exactly once.


Ownership questions you should always ask:

  • Who allocated this memory?

  • Who must delete it?

  • Is it shared?

  • Could two pointers refer to the same block?


This thinking prevents many bugs.

8) A tiny example with a class

class Box {
public:
    Box() {
        std::cout << "Box created\n";
    }

    ~Box() {
        std::cout << "Box destroyed\n";
    }
};

int main() {
    Box *b = new Box();
    delete b;
}

What happens:

  1. new Box() allocates memory

  2. constructor runs

  3. delete b calls destructor

  4. memory is freed


This is the C++ object model in action.

9) Key difference between C and C++


Here is the clean mental summary:


C

  • malloc / free

  • raw memory only

  • no constructors/destructors


C++

  • new / delete

  • object construction and destruction

  • correct for classes


That is why in C++ you should not casually replace new with malloc.

10) What to remember forever


Here is the core rule set:

  • new allocates one object on the heap

  • new[] allocates an array on the heap

  • delete frees one object

  • delete[] frees an array

  • new/delete must match

  • new[]/delete[] must match

  • after delete, the pointer becomes dangling

  • malloc/free are C functions and do not handle constructors/destructors

11) The beginner trap to avoid


Many beginners think:


“Pointer variable is on the stack, so the data is also on the stack.”


No. The pointer is only an address.

The actual data may be on the heap.


That distinction is one of the most important ideas in C++ memory management.

12) Small mental test for you


Consider this:

int *p = new int(100);
int *q = p;
delete p;

Now ask yourself:

  • Does q still point to the same address?

  • Is that address valid?

  • What would happen if you wrote *q?


That is exactly how you build strong intuition.
